Decision Guide — Which Should You Choose?

The right answer depends on your project type and workflow style.

?

Choose BMAD when…

You're building a multi-phase product and want structured agile ceremonies with AI. You need specialized roles — a PM to write specs, an architect to design systems, a scrum master to manage sprints. Great for teams that already use agile and want to supercharge it with AI. Also excellent when you want Party Mode — multiple AI experts debating the right design.

?

Choose GSD when…

You're a solo developer or small team who wants to describe an idea and have it reliably built — without context degradation killing quality mid-way. If you've ever had Claude "forget" what you were building or produce inconsistent code across a long session, GSD solves this with fresh context per execution wave. Best for spec-first, reliability-focused projects.

?

Choose Superpowers when…

You want your coding agent to simply be better without you learning new commands or workflows. Superpowers is the lowest-friction option — install it and your agent automatically practices TDD, does systematic debugging, creates git worktrees, and reviews its own code. The zero-ceremony choice that works for any project type.

∑

Use All Three Together

Power users combine them. BMAD for planning and architecture (PRD, tech spec, sprint planning) → GSD for reliable execution (wave-based implementation with fresh context) → Superpowers running always-on underneath (TDD, code review, debugging). Each layer handles a different concern without conflict.

How to Use Effectively with Claude Code

Regardless of which framework you choose, these principles maximize your Claude Code results.

1

Install in the Right Scope

Install globally (--global) if you want the framework on all projects. Install locally (--local) for project-specific configs. BMAD and GSD both support per-project customization.

2

Run Claude Code with --dangerously-skip-permissions

All three frameworks work best when Claude isn't constantly asking you to approve bash commands like git commit or ls. Add granular permissions if you prefer control, but skip-permissions mode is the intended workflow for automation-heavy usage.

3

Front-Load Your Context

All three frameworks work better when Claude has rich project context. Describe your stack, constraints, and goals at the start. BMAD's agents will ask clarifying questions. GSD's /gsd-new-project will extract them systematically. Superpowers will brainstorm with you. Don't skip the design phase.

4

Let the Framework Drive — Don't Micromanage

The biggest mistake is overriding the framework mid-flow with ad-hoc instructions. Trust the process: BMAD's agents have expertise; GSD's wave execution is designed for reliability; Superpowers skills auto-activate for a reason. Intervene at review checkpoints, not during execution.

5

Keep CLAUDE.md / Project Context Updated

Claude Code reads CLAUDE.md at the start of every session. Put your architecture decisions, coding standards, and framework preferences there. BMAD, GSD, and Superpowers all benefit from a well-maintained project context file.

◆ Get Shit Done — Complete Guide

Context Engineering for
Reliable AI Development

GSD fixes "context rot" — the quality degradation that happens as Claude fills its context window. 58k stars. 86 skills. 33 subagents. Multi-agent wave execution. The most battle-tested spec-driven system for Claude Code.

The Problem GSD Solves

Without GSD, long Claude Code sessions produce increasingly inconsistent results. Context fills up, early instructions get compressed, and Claude starts making decisions you didn't ask for.

⚠️ Context Rot: As your session grows, Claude's context window fills. Early instructions get deprioritized, quality degrades, and you get "I'll be more concise now" — a symptom that the model is running out of reliable working memory.
✨ GSD's fix: Each execution plan runs in a fresh 200k context window. Your main session stays lean (30-40% usage). Sub-agents do the heavy lifting with pristine context — no accumulated garbage.

Discuss Phase — Shape the Implementation

3

Plan Phase — Research, Plan, Verify

GSD researches how to implement the phase (guided by your CONTEXT.md), creates 2-3 atomic XML-structured plans, then verifies they achieve the phase goals. If not, it loops until they pass.

Claude Code
/gsd-plan-phase 1

Creates: 1-RESEARCH.md, 1-1-PLAN.md, 1-2-PLAN.md, …

Example XML plan structure
<task type="auto">
  <name>Create login endpoint</name>
  <files>src/app/api/auth/login/route.ts</files>
  <action>
    Use jose for JWT (not jsonwebtoken - CommonJS issues).
    Validate credentials against users table.
    Return httpOnly cookie on success.
  </action>
  <verify>curl -X POST localhost:3000/api/auth/login returns 200</verify>
  <done>Valid credentials return cookie, invalid return 401</done>
</task>
4

Plans group into waves by dependency. Independent plans run in parallel, each with a fresh 200k context. Your main session stays at 30-40%. Every task gets an atomic git commit.

Git Worktrees Skill — Isolated Development

After design approval, the agent creates a git worktree on a new branch. This isolates your feature from main, runs project setup, and verifies you have a clean test baseline before touching anything.

What happens automatically

3

Writing Plans Skill — Atomic Implementation Tasks

The agent breaks work into 2-5 minute tasks. Each task has exact file paths, complete expected code, and verification steps. Plans are detailed enough for "an enthusiastic junior engineer with poor judgement" to follow — nothing ambiguous.

Example plan task
Task 1: Create rate limiter class
  File: src/middleware/rateLimiter.ts
  Action: Create RateLimiter class with Map storage,
          increment(), isLimited(), and cleanup() methods.
  Verify: Unit test passes — 5 requests succeed, 6th blocked
  Time: ~3 minutes
4

TDD Skill — Enforced RED-GREEN-REFACTOR

The TDD skill is the most rigorous in Superpowers. It enforces the cycle strictly: write a failing test, watch it fail (RED), write minimal code to pass (GREEN), refactor, commit. Code written before tests gets deleted.

TDD cycle enforced automatically
# 1. Write failing test FIRST:
test('blocks 6th request within window', () => {
  // This test FAILS first — that's correct (RED)
  const limiter = new RateLimiter(5, 60);
  for (let i = 0; i < 5; i++) limiter.increment('ip');
  expect(limiter.isLimited('ip')).toBe(true);
});

# 2. Watch it fail — confirms test is real (RED ✓)
npm test  → FAIL (expected!)

# 3. Write minimal code to pass (GREEN)
# 4. Refactor if needed
# 5. Commit
5

Subagent-Driven Development — Two-Stage Review

Tasks are dispatched to fresh subagents. After implementation, each task undergoes a two-stage review: first for spec compliance (does it do what was planned?), then for code quality (is it well-written?). Critical issues block progress.

6

Finishing a Branch — Merge/PR Decision Workflow

When all tasks complete, the finishing skill verifies tests pass end-to-end, then presents options: merge to main, create PR, keep branch for more work, or discard. Cleans up the worktree.

Combine with GSD for Maximum Reliability

Run Superpowers on top of GSD. GSD manages context and wave execution at the macro level. Superpowers adds TDD, code review, and debugging methodology at the micro level. The skills don't conflict — they complement each other perfectly.

Combined stack
# Install both
npx get-shit-done-cc --claude --global     # GSD
/plugin install superpowers@claude-plugins-official  # Superpowers

# GSD handles: planning, context, wave execution, milestones
# Superpowers handles: TDD within each plan, code review, debugging
# Result: structured development + quality methodology, no gaps
💡

Let It Run — Don't Interrupt

Superpowers is designed for autonomous multi-hour runs. Intervene only at natural checkpoints: design approval, plan review, and the merge/PR decision. Mid-task interruptions break the skill's state machine.
